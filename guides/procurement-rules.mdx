When a Stock Request is confirmed, Odoo's procurement engine evaluates rules to determine how to fulfill the request. This guide explains how the process works.

## How Procurement Works

The procurement process follows these steps:

<Steps>
  <Step title="Request Confirmation">
    User confirms a Stock Request, triggering the `action_confirm()` method
  </Step>
  
  <Step title="Procurement Launch">
    The system calls `_action_launch_procurement_rule()` to start the procurement process
  </Step>
  
  <Step title="Rule Evaluation">
    Odoo searches for applicable procurement rules based on:
    - Destination location
    - Product routes
    - Warehouse routes
    - Manually selected route (if any)
  </Step>
  
  <Step title="Action Execution">
    The matching rule executes its action:
    - `Pull From` → Creates stock move/transfer
    - `Buy` → Creates purchase order
    - `Manufacture` → Creates manufacturing order
  </Step>
  
  <Step title="Allocation Creation">
    Stock Request Allocation links the request to generated moves
  </Step>
</Steps>

## Procurement Rule Evaluation

### Route Computation

The system determines available routes for a request:

```python
@api.depends("product_id", "warehouse_id", "location_id")
def _compute_route_ids(self):
    route_obj = self.env["stock.route"]
    # Get warehouse routes
    routes = route_obj.search(
        [("warehouse_ids", "in", self.mapped("warehouse_id").ids)]
    )
    
    for record in self:
        routes = route_obj
        # Add product and category routes
        if record.product_id:
            routes += record.product_id.mapped("route_ids") | \
                      record.product_id.mapped("categ_id").mapped("total_route_ids")
        
        # Add warehouse routes
        if record.warehouse_id:
            routes |= routes_by_warehouse[record.warehouse_id.id]
        
        # Filter by destination location
        parents = record.get_parents().ids
        record.route_ids = routes.filtered(
            lambda r: any(p.location_dest_id.id in parents for p in r.rule_ids)
        )
```

Routes are computed from:
1. **Product routes**: Routes configured on the product form
2. **Category routes**: Routes from the product category
3. **Warehouse routes**: Routes assigned to the warehouse
4. **Filtered by location**: Only rules that deliver to the request location (or parent locations)

### Manual Route Selection

Users can force a specific route:

```python
route_id = fields.Many2one(
    "stock.route",
    string="Route",
    domain="[('id', 'in', route_ids)],"
    ondelete="restrict",
)
```

The domain ensures only applicable routes can be selected.

## Procurement Execution

The main procurement method:

```python
def _action_launch_procurement_rule(self):
    """
    Launch procurement group run method with required/custom fields
    generated by a stock request. Procurement group will launch
    '_run_move', '_run_buy' or '_run_manufacture' depending on the
    stock request product rule.
    """
    errors = []
    for request in self:
        if request._skip_procurement():
            continue
        
        # Check if already fulfilled
        qty = sum(move.product_qty for move in request.move_ids.filtered(
            lambda r: r.state != "cancel"
        ))
        if qty >= request.product_qty:
            continue
        
        # Check stock availability first
        if request.company_id.stock_request_check_available_first:
            if request.product_id.with_context(
                location=request.location_id.id
            ).free_qty >= request.product_uom_qty:
                request._action_use_stock_available()
                continue
        
        # Prepare procurement values
        values = request._prepare_procurement_values(
            group_id=request.procurement_group_id
        )
        
        # Run procurement
        try:
            procurements = [
                self.env["procurement.group"].Procurement(
                    request.product_id,
                    request.product_uom_qty,
                    request.product_uom_id,
                    request.location_id,
                    request.name,
                    request.name,
                    self.env.company,
                    values,
                )
            ]
            self.env["procurement.group"].run(procurements)
        except UserError as error:
            errors.append(str(error))
    
    if errors:
        raise UserError("\n".join(errors))
    return True
```

### Skip Conditions

Procurement is skipped when:

```python
def _skip_procurement(self):
    return self.state != "draft" or self.product_id.type not in ("consu", "product")
```

- Request is not in draft state
- Product is a service (only storable products and consumables can be procured)

## Using Available Stock First

If configured, the system checks for available stock before running procurement:

```python
def _action_use_stock_available(self):
    """Create a stock move with the necessary data and mark it as done."""
    allocation_model = self.env["stock.request.allocation"]
    stock_move_model = self.env["stock.move"].sudo()
    
    quants = self.env["stock.quant"]._gather(self.product_id, self.location_id)
    pending_qty = self.product_uom_qty
    
    for quant in quants.filtered(lambda x: x.available_quantity >= 0):
        qty_move = min(pending_qty, quant.available_quantity)
        if qty_move > 0:
            move = stock_move_model.create(self._prepare_stock_move(qty_move))
            move._action_confirm()
            pending_qty -= qty_move
            
            # Create allocation + done move
            allocation_model.create(
                self._prepare_stock_request_allocation(move)
            )
            move.quantity = move.product_uom_qty
            move.picked = True
            move._action_done()
```

This creates immediate moves for available quantities, avoiding unnecessary procurement.

<Note>
Enable this feature in **Settings → Stock Requests → Check Available Stock First**
</Note>

## Procurement Values

The system passes custom values to procurement rules:

```python
def _prepare_procurement_values(self, group_id=False):
    """Prepare specific key for moves or other components that
    will be created from a procurement rule coming from a stock
    request. This method could be overridden to add other custom
    keys that could be used in move/po creation.
    """
    return {
        "date_planned": self.expected_date,
        "warehouse_id": self.warehouse_id,
        "stock_request_allocation_ids": self.id,
        "group_id": group_id or self.procurement_group_id.id or False,
        "route_ids": self.route_id,
        "stock_request_id": self.id,
    }
```

These values are passed to rules and used to:
- Schedule transfers for the expected date
- Link moves back to the request via allocations
- Associate with the procurement group
- Respect the selected route

## Stock Move Generation

When a procurement rule creates a stock move, the request ID is embedded:

```python
# In stock.rule model
def _get_stock_move_values(
    self,
    product_id,
    product_qty,
    product_uom,
    location_id,
    name,
    origin,
    company_id,
    values,
):
    result = super()._get_stock_move_values(
        product_id, product_qty, product_uom,
        location_id, name, origin, company_id, values
    )
    
    if values.get("stock_request_id", False):
        result["allocation_ids"] = [
            (0, 0, {
                "stock_request_id": values.get("stock_request_id"),
                "requested_product_uom_qty": product_qty,
            })
        ]
    return result
```

This creates the Stock Request Allocation that links the move to the request.

## Transfer Generation Example

Consider a request for 10 units of Product A to Location/Stock:

**Scenario: Pull from Vendor Location**

1. Request confirmed for 10 units to WH/Stock
2. System finds route: "Buy" with rule "Vendor → WH/Stock"
3. Rule action is "Buy", so it creates a Purchase Order
4. When PO is confirmed, incoming move is created
5. Incoming move is linked to request via allocation
6. When received, request qty_done updates automatically

**Scenario: Pull from Another Warehouse**

1. Request confirmed for 10 units to WH/Stock
2. System finds route: "Transfer from WH2"
3. Rule action is "Pull From", creates internal transfer
4. Transfer: WH2/Stock → WH/Stock
5. Transfer linked to request via allocation
6. When transferred, request is fulfilled

  Configure routes on products that are always procured the same way (e.g., always manufactured or always purchased).

## Debugging Procurement Issues

If procurement fails:

<Steps>
  <Step title="Check Route Availability">
    Verify that applicable routes exist for the product and destination location:
    - Check product routes on product form
    - Check warehouse routes in warehouse configuration
    - Ensure routes have rules with destination matching the request location
  </Step>
  
  <Step title="Verify Rule Configuration">
    Ensure rules are properly configured:
    - Action type (Pull From, Buy, Manufacture)
    - Source and destination locations
    - Procurement route assignment
  </Step>
  
  <Step title="Check Product Configuration">
    Verify product settings:
    - Product type is Storable or Consumable
    - Product is not archived
    - Product has valid vendor if using Buy rules
    - Product has valid BoM if using Manufacture rules
  </Step>
  
  <Step title="Review Error Messages">
    The system raises UserError with details when procurement fails. Check:
    - No rule found for the product/location combination
    - Missing vendor configuration
    - Missing BoM for manufacturing
  </Step>
</Steps>

<Warning>
If no applicable rule is found, the system will raise an error: "No rule found to fulfill the procurement for Product X to Location Y"
</Warning>

## Integration with Procurement Group

Requests can share a procurement group:

```python
procurement_group_id = fields.Many2one(
    "procurement.group",
    "Procurement Group",
    help="Moves created through this stock request will be put in this "
    "procurement group. If none is given, the moves generated by "
    "procurement rules will be grouped into one big picking.",
)
```

Benefits:
- Groups related moves into same transfer
- Easier tracking of project materials
- Consolidated reporting
