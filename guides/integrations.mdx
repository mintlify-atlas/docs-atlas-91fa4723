## Purchase Orders

The `stock_request_purchase` module enables visibility into purchase orders generated from stock requests.

### Features

- View purchase orders created from stock requests
- Link purchase order lines to originating requests
- Track procurement through the purchase lifecycle
- Automatic cancellation propagation

### Installation

```bash
odoo-bin -u stock_request_purchase
```

### How It Works

When a stock request is confirmed and the procurement rule action is "Buy":

1. Purchase order is created by the procurement engine
2. Purchase order lines are linked to the stock request
3. A "Purchase Orders" smart button appears on the request
4. Users can navigate between requests and purchase orders

### Viewing Purchase Orders from Requests

```python
class StockRequest(models.Model):
    _inherit = "stock.request"

    purchase_ids = fields.One2many(
        "purchase.order",
        compute="_compute_purchase_ids",
        string="Purchase Orders",
        readonly=True,
    )
    purchase_count = fields.Integer(
        compute="_compute_purchase_ids",
        readonly=True
    )
    purchase_line_ids = fields.Many2many(
        "purchase.order.line",
        string="Purchase Order Lines",
        readonly=True,
        copy=False
    )

    def action_view_purchase(self):
        action = self.env["ir.actions.act_window"]._for_xml_id(
            "purchase.purchase_rfq"
        )
        purchases = self.mapped("purchase_ids")
        if len(purchases) > 1:
            action["domain"] = [("id", "in", purchases.ids)]
        elif purchases:
            action["views"] = [
                (self.env.ref("purchase.purchase_order_form").id, "form")
            ]
            action["res_id"] = purchases.id
        return action
```

### Viewing Requests from Purchase Orders

```python
class PurchaseOrder(models.Model):
    _inherit = "purchase.order"

    stock_request_ids = fields.Many2many(
        comodel_name="stock.request",
        string="Stock Requests",
        compute="_compute_stock_request_ids",
    )
    stock_request_count = fields.Integer(
        "Stock Request #",
        compute="_compute_stock_request_ids"
    )

    @api.depends("order_line")
    def _compute_stock_request_ids(self):
        for rec in self:
            rec.stock_request_ids = rec.order_line.mapped("stock_request_ids")
            rec.stock_request_count = len(rec.stock_request_ids)

    def action_view_stock_request(self):
        action = self.env["ir.actions.act_window"]._for_xml_id(
            "stock_request.action_stock_request_form"
        )
        requests = self.mapped("stock_request_ids")
        if len(requests) > 1:
            action["domain"] = [("id", "in", requests.ids)]
        elif requests:
            action["views"] = [
                (self.env.ref("stock_request.view_stock_request_form").id, "form")
            ]
            action["res_id"] = requests.id
        return action
```

### Cancellation Propagation

When a request is cancelled, related purchase orders can also be cancelled:

```python
def action_cancel(self):
    """Propagate the cancellation to the generated purchase orders."""
    res = super().action_cancel()
    if not self.env.context.get("skip_cancel_po_from_stock_request"):
        self.sudo().purchase_ids.filtered(
            lambda x: x.state not in ("purchase", "done", "cancel")
            and x.stock_request_ids == self
        ).button_cancel()
    return res
```

<Warning>
Only RFQs and draft POs are cancelled. Confirmed purchase orders must be cancelled manually.
</Warning>

## Manufacturing Orders

The `stock_request_mrp` module integrates stock requests with manufacturing orders.

### Features

- View manufacturing orders created from stock requests
- Link MOs to originating requests
- Track production through manufacturing lifecycle
- Support for make-to-order scenarios

### Installation

```bash
odoo-bin -u stock_request_mrp
```

### How It Works

When a stock request is confirmed and the procurement rule action is "Manufacture":

1. Manufacturing order is created by the procurement engine
2. MO is linked to the stock request
3. A "Manufacturing Orders" smart button appears on the request
4. Finished products are allocated to the request

### Viewing Manufacturing Orders from Requests

```python
class StockRequest(models.Model):
    _inherit = "stock.request"

    production_ids = fields.Many2many(
        "mrp.production",
        "mrp_production_stock_request_rel",
        "stock_request_id",
        "mrp_production_id",
        string="Manufacturing Orders",
        readonly=True,
        copy=False,
    )
    production_count = fields.Integer(
        string="Manufacturing Orders count",
        compute="_compute_production_ids",
        readonly=True,
    )

    def action_view_mrp_production(self):
        action = self.env["ir.actions.act_window"]._for_xml_id(
            "mrp.mrp_production_action"
        )
        productions = self.mapped("production_ids")
        if len(productions) > 1:
            action["domain"] = [("id", "in", productions.ids)]
        elif productions:
            action["views"] = [
                (self.env.ref("mrp.mrp_production_form_view").id, "form")
            ]
            action["res_id"] = productions.id
        return action
```

### Finished Product Allocation

When the MO produces finished goods, they are allocated to the request:

```python
def _get_move_finished_values(
    self,
    product_id,
    product_uom_qty,
    product_uom,
    operation_id=False,
    byproduct_id=False,
    cost_share=0,
):
    res = super()._get_move_finished_values(
        product_id, product_uom_qty, product_uom,
        operation_id=operation_id,
        byproduct_id=byproduct_id,
        cost_share=cost_share,
    )
    if self.stock_request_ids:
        res["allocation_ids"] = [
            (0, 0, {
                "stock_request_id": request.id,
                "requested_product_uom_qty": request.product_qty,
            })
            for request in self.stock_request_ids
        ]
    return res
```

## Bill of Materials

The `stock_request_bom` module enables creating requests for all BOM components at once.

### Features

- Select a BOM on the Stock Request Order
- Automatically generate request lines for all BOM components
- Specify quantity multiplier for the BOM
- Update or clear BOM selections

### Installation

```bash
odoo-bin -u stock_request_bom
```

### How It Works

1. Create a Stock Request Order
2. Select a BOM product and quantity
3. System auto-fills request lines with all BOM components
4. Quantities are calculated based on BOM quantity multiplier
5. Confirm the order to process all components

### Usage

<Steps>
  <Step title="Create Order">
    Navigate to Stock Requests → Stock Request Orders → Create
  </Step>
  
  <Step title="Select BOM Product">
    In the **Product BOM** field, choose a product that has a configured Bill of Materials
  </Step>
  
  <Step title="Set BOM Quantity">
    Enter the **Quantity BOM** - this is the multiplier for the BOM components
    
    For example, if BOM defines 2 units of Component A and you set quantity to 5,
    the request will be for 10 units of Component A
  </Step>
  
  <Step title="Auto-Fill Components">
    The system automatically creates request lines for each BOM component with calculated quantities
  </Step>
  
  <Step title="Adjust and Confirm">
    Review the generated lines, make any adjustments, and confirm the order
  </Step>
</Steps>

<Tip>
This is useful for kitting operations or when requesting all materials needed to assemble a product.
</Tip>

## Kanban Cards

The `stock_request_kanban` module implements lean manufacturing kanban card workflows with barcode scanning.

### Features

- Print kanban cards with QR codes for products
- Scan cards to create stock requests
- Batch scanning for multiple cards
- Configure CRC check digits and barcode formats

### Installation

```bash
odoo-bin -u stock_request_kanban
```

### Configuration

Access Settings → Technical → System Parameters:

**CRC Check Digit**
- Parameter: `stock_request_kanban.crc`
- Value: `0` to disable CRC, `1` to enable
- Default: Enabled

**Ignore CRC**
- Parameter: `stock_request_kanban.ignore_crc`
- Value: `1` to ignore CRC validation

**Barcode Format**
- Parameter: `stock_request_kanban.barcode_format`
- Value: Custom format string for QR codes

### Creating Kanban Cards

<Steps>
  <Step title="Navigate to Kanban">
    Go to Stock Requests → Stock Requests Kanban → Create
  </Step>
  
  <Step title="Configure Card">
    - Select the product
    - Enter the quantity per card
    - Choose the destination location
  </Step>
  
  <Step title="Save and Print">
    Save the kanban record and print the card with QR code
  </Step>
  
  <Step title="Place Card">
    Put the printed card in the product storage location
  </Step>
</Steps>

### Ordering via Kanban

**Single Card Method:**

<Steps>
  <Step title="Consume Product">
    When the product is consumed, take the kanban card
  </Step>
  
  <Step title="Scan Card">
    Go to Stock Requests → Order Kanban Card and scan the card
  </Step>
  
  <Step title="Request Created">
    The system automatically creates a stock request for the product and quantity on the card
  </Step>
</Steps>

**Batch Scanning Method:**

<Steps>
  <Step title="Collect Cards">
    As products are consumed, collect the kanban cards in a storage location
  </Step>
  
  <Step title="Create Order">
    Create a new Stock Request Order
  </Step>
  
  <Step title="Scan Multiple Cards">
    Click the scan button and scan all collected cards
  </Step>
  
  <Step title="Review and Confirm">
    Review the auto-generated request lines and confirm the order
  </Step>
</Steps>

<Note>
Batch scanning is ideal for scheduled replenishment rounds, while single card scanning supports just-in-time workflows.
</Note>

## Integration Comparison

| Feature | Purchase | MRP | BOM | Kanban |
|---------|----------|-----|-----|--------|
| **Purpose** | Link to POs | Link to MOs | Auto-create components | Barcode workflows |
| **Automation** | View generated POs | View generated MOs | Auto-fill from BOM | Scan to create |
| **Direction** | Request → PO | Request → MO | BOM → Requests | Card → Request |
| **Use Case** | Procurement tracking | Production tracking | Kitting/Assembly | Lean inventory |
| **Dependencies** | Purchase module | MRP module | MRP module | None |

## Combining Integrations

You can use multiple integration modules together:

**Example: Kanban + Purchase**
- Scan kanban card for purchased product
- Request auto-created
- Procurement generates PO
- Track PO from request

**Example: BOM + MRP**
- Create order from BOM for final product components
- Confirm order
- Some components purchased, others manufactured
- Track POs and MOs from respective requests

**Example: All Modules**
- Scan kanban card for assembled product
- Create request from card
- Request fulfilled via manufacturing
- Manufacturing order components trigger purchase
- Full visibility from card scan to component purchase to assembly
